# 视频翻译工作流说明书

这是一套用于本地视频高质量中文字幕制作的工作流，目标是用词级时间轴、AI 语义分段、专业术语修复和最终质检，把普通机器字幕提升到接近人工字幕的观看效果。

作者 / 工作流设计：`AI落地第四声`

作者声明：作者 / 工作流设计为 `AI落地第四声`。本作者信息用于展示和来源识别，不添加额外授权限制。

本说明书写给人类使用者和维护者阅读，帮助快速理解这套工作流适合做什么、为什么效果好，以及使用前需要准备什么。AI、WorkBuddy 或自动化流程应阅读英文版执行规则，不要把本中文说明书当作执行规则。

## 使用前需要准备什么

1. 去 OkFile 注册并创建 API Key：`https://www.okfile.com/`。API Key 页面通常在 `https://www.okfile.com/en/account/api-keys`。OkFile 用于把本地音频临时上传成可访问链接，方便阿里 Fun-ASR 转写。
2. 去阿里百炼 / DashScope 注册并创建 API Key：`https://bailian.console.aliyun.com/`。建议先充值少量余额，例如 2-10 元，用于调用阿里 Fun-ASR 转写。
3. 准备好 `阿里云 API Key`、`工作空间 ID`、`OkFile API Key`，以及要翻译的视频本地路径。
4. 把视频本地路径复制给 AI，让 AI 使用这个 skill 翻译字幕即可。首次运行会先检查本地环境和必备工具，并提示你补充 `.env` 文件中的信息。

首次使用时，AI 应直接说明：这套工作流会在本地提取音频，通过 OkFile 生成临时链接，用阿里 Fun-ASR 做词级转写，再固定调用阿里 `qwen-mt-plus` 生成分段翻译，最后由当前 AI agent 做时间轴匹配、术语修复、质检和 ASS/SRT 导出。当前固定模型组合是多轮视频测试后效果和性价比最稳的方案：转写固定用 Fun-ASR，分段翻译固定用 `qwen-mt-plus`；如果 agent 是 WorkBuddy，编排模型建议 DeepSeek V4 Pro；如果是 Codex / Cursor，编排模型建议 GPT 5.5 级模型。

你只需要准备 3 个核心信息：`DASHSCOPE_API_KEY`、`ALIYUN_WORKSPACE_ID`、`OKFILE_TOKEN`。准备好后写入 `.env`，再把本地视频路径发给 AI 即可。

## 固定模型方案

从 v1.3.0 开始，普通用户不再需要在多个模型之间选择。原因很简单：我们已经测试过多条真实视频，很多模型在长任务、分段、格式遵守和续跑稳定性上不够可靠，给用户太多选择反而增加失败率。

当前固定方案：

- 转写模型：阿里 Fun-ASR。它支持长音频和词级时间戳，是字幕时间轴准确的基础。
- 分段翻译模型：阿里 `qwen-mt-plus`。它通过内置 helper 批量生成 `segments.txt`，支持缓存和续跑。生产默认使用串行模式，避免并发过高触发限流；质量和稳定性优先于绝对速度。
- 编排模型：WorkBuddy 场景推荐 DeepSeek V4 Pro；Codex / Cursor 场景推荐 GPT 5.5 级模型。编排模型不负责大批量翻译正文，主要负责执行流程、匹配时间轴、质检修复、导出字幕和向用户汇报结果。

除非用户明确要求重构方案，否则不要临时切换 ASR、helper 或本地模型。

## 如何使用

正常使用时，人不需要手动处理每个脚本。你只需要把本地视频交给 WorkBuddy、Codex 或其他支持该 skill 的自动化工具，并在任务开始前确认几个问题：

> 简单说：把本地视频路径发给 AI，回答几个确认问题，剩下由 AI 跑流程。命令行是 AI 的后台执行方式，不是普通用户必须记住的使用方式。

- 源语言是否默认英文，是否需要改成法语、西班牙语或意大利语。
- 目标语言是否保持中文。当前默认目标语言为中文，内置术语库、排版和质检规则也主要围绕中文字幕优化；如果要稳定翻译成英文、法语、西班牙语、意大利语等目标语言，需要补充对应目标语言的提示词、术语库和质检规则。
- 视频画面里是否有大量文字、PPT、图表、软件界面、代码、指标名，或有意义但旁白没有完整讲出来的图片信息。
- 是否启用 OCR/屏幕上下文辅助。默认不启用；只有画面文字确实影响理解时才建议启用，因为它会增加处理时间和成本。
- 字幕导出路径是否使用项目默认 `outputs/` 文件夹。如果要导出到其他位置，直接提供新的文件夹路径即可。

流程完成后，AI 会在对话框里返回本次任务摘要，包括输出文件、总耗时、各环节耗时、转写模型、翻译模型、编排模型、QA warning 概况等。随后会输出两类字幕文件：

- ASS：推荐观看格式，能固定中英文字号、位置和样式。
- SRT：通用兼容格式，适合上传平台或备用。

文件名会保留原视频名，并加上类似 `.zh-en`、`.zh-fr` 的语言标识。

## 长视频处理时的状态反馈

如果视频超过 10 分钟，AI 在开始前应先告诉你：视频大约多长、预计首轮处理大概需要多久、以及每 10 分钟会反馈一次状态。这个预计只能给范围，不能保证精确，因为上传速度、阿里排队、AI 模型速度、是否启用屏幕上下文、是否需要 QA 修复都会影响耗时。

长任务执行中，AI 不应该长时间沉默，也不应该说“等通知”然后结束当前轮次。WorkBuddy 这类平台的通知可能不会自动唤醒 agent，所以正确做法是让任务在前台持续执行，或在同一轮次里循环读取任务状态直到完成、暂停或失败。每 10 分钟至少应反馈一次：

- 已完成哪些环节。
- 当前正在执行什么环节。
- 是否正在复用缓存。
- 如果能判断，粗略还剩多久。

每个任务结束后，运行摘要会记录实际耗时。后续处理同类视频时，应参考这些历史耗时，让预估逐渐更接近真实情况。

## 常见使用场景

以下场景可以直接使用这套工作流：

- 课程、培训、讲解视频：适合把英文或小语种视频做成自然的中文字幕。
- 金融交易、投资、期权、订单流视频：内置术语库对这类内容最友好，能避免明显教材腔或错译。
- 访谈、播客、分享、市场复盘：如果主要信息来自人声，通常不需要启用 OCR/屏幕上下文。
- 录屏课程、软件教程、PPT 讲解：如果屏幕文字、图表、按钮、代码或指标名会影响理解，建议启用屏幕上下文。
- 批量视频：适合按批次处理，但建议先抽查 1-2 个成品，确认术语和字幕节奏符合预期后再继续批量跑。
- 非金融领域视频：可以使用同一流程，但应维护对应领域的术语库、专名和表达风格。

以下场景可以处理，但需要降低自动化预期：

- 音频噪声很大、强口音明显、多人同时说话的视频。
- 画面信息很重要，但用户不愿启用屏幕上下文的视频。
- 需要翻译成非中文目标语言的视频。当前流程架构可以扩展，但默认内置规则仍以中文字幕为主。

## 这套工作流做什么

这套工作流用于把本地视频快速制作成自然、专业、接近人工字幕质量的中文字幕。它不是简单把字幕丢给 AI 翻译，而是把“听写识别、语义分段、专业翻译、时间轴匹配、排版和质检”拆成一套完整流程。

它特别适合课程、培训、讲解、分享、访谈、市场复盘、软件教学等内容。目前主要用于金融交易类视频，因此内置术语库也优先服务交易、投资、订单流、期权和市场分析等内容。

## 工作流概述

整体流程可以概括为：

```text
本地视频 -> 提取音频 -> 上传 OkFile -> 阿里 Fun-ASR 词级转写 -> qwen-mt-plus 分段翻译 -> 术语修复 -> 时间轴匹配 -> 质检 -> 导出 ASS/SRT
```

可选 OCR/屏幕上下文只作为补充信息使用，不替代语音转写和词级时间轴。它适合用于 PPT、图表、软件界面、代码演示、屏幕文字较多的视频。普通口播或画面文字不重要的视频，不建议启用。

## 为什么效果更好

普通字幕工具通常直接依赖转写模型的原始断句。原始断句经常不符合中文观看习惯，也容易出现长句堆叠、断点奇怪、术语生硬和中英文不匹配的问题。

这套工作流的关键差异是：不直接相信机器原始断句，而是让 AI 基于完整上下文重新分段和翻译，再用词级时间戳回填时间轴。这样字幕会更像人工处理过的结果，而不是机器转写的副产品。

## 核心优势

- 词级时间戳对齐，说话节奏和字幕显示更自然。
- AI 语义分段，减少机器断句带来的生硬感。
- 中文表达优先按观看体验处理，而不是逐词硬翻。
- 可维护术语库，让不同领域的译法逐渐稳定。
- 支持 ASS 样式控制，中文更清楚，源语言辅助展示。
- 可选屏幕上下文辅助，帮助理解 PPT、图表、软件界面和屏幕文字。
- 最终带质检流程，尽量减少漏词、长句、闪字幕、孤立碎片和术语错误。
- 不需要下载本地大模型，转写速度更快，也更节省本地硬盘和算力。

## 适合什么视频

当前最适合：

- 英文课程和培训视频。
- 金融交易、投资、订单流、价格行为、期权、宏观市场类讲解。
- 录屏课程、webinar、软件平台教学。
- 访谈、分享、市场复盘、策略讲解。

也可以处理少量常见小语种视频：

- 法语
- 西班牙语
- 意大利语

这套流程不局限于金融交易。如果长期用于电影、法律、医学、游戏、产品教程等场景，只要持续维护对应领域的术语库和表达风格，也可以逐渐适配其他内容类型。

## 流程能力边界

- 当前服务录制好的本地视频，不做实时字幕或直播同传。
- 当前导出 ASS/SRT 字幕文件，不负责把字幕硬压进视频。
- 当前默认目标语言为中文，内置流程、术语库、热词库和质检规则都优先围绕“翻译成中文”优化。
- 工作流架构可以扩展到英文、法语、西班牙语、意大利语等目标语言，但需要为目标语言补充提示词、术语库、排版和质检规则，不能直接假设和中文路线一样成熟。
- 当前生产主线固定为 OkFile + 阿里 Fun-ASR。
- OCR/屏幕上下文默认关闭，只在视觉信息确实重要时启用。
- 如果没有可靠的词级时间戳，字幕时间轴质量会明显下降。
- 如果术语库不维护，跨领域翻译质量会下降。
- 低质量音频、强口音、多人重叠对话、背景噪声很大的视频，转写识别准确性会显著降低，仍建议人工抽查。

## 需要的本地环境和工具

本地需要：

- Python 3.9 或更高版本。
- `ffmpeg`，用于从视频里提取音频，也用于可选截图。
- 可访问 OkFile 和阿里百炼 / DashScope 的网络。
- 足够的本地空间，用于临时音频、转写结果、提示词、字幕文件和质检报告。

第一次在新设备上运行时，AI 会先检查这些环境和工具。如果缺少 `ffmpeg` 等工具，AI 会提示安装方式，或在需要时请求你的授权后再安装。检查通过后，后续同一设备通常不需要重复做这一步。

## 为什么选择阿里 Fun-ASR

这套流程选择阿里 Fun-ASR，主要因为它适合长录音转写，支持异步 API，并能返回词级时间戳。词级时间戳是整套工作流的关键，因为后续所有字幕分段都要精确匹配回真实语音时间。

Fun-ASR 还可以配置热词和词表，适合提升人名、品牌名、指标名、平台名和专业术语的识别效果。

当前参考价格约为：

- `0.00022 元/秒`
- `0.0132 元/分钟`
- `0.792 元/小时`

实际价格和免费额度可能变化，批量处理前应以阿里百炼官方价格页为准。

## 为什么选择 OkFile

Fun-ASR 需要访问在线音频 URL，而源视频通常在本地电脑上。OkFile 的作用是把本地提取出的音频临时上传成可访问链接，再交给 Fun-ASR 转写。

选择 OkFile 的原因是轻量、注册方便、能通过 API 上传文件并生成 URL。对这套工作流来说，它比自行配置 OSS、签名 URL 或云存储权限更简单。

## 为什么不使用本地大模型

本地跑 Whisper 等大模型会占用硬盘、CPU/GPU 和较长时间。对视频字幕工作流来说，本地电脑更适合做轻量任务：提取音频、组织文件、匹配时间轴、生成字幕和质检。

真正耗时的转写交给云端 API，整体更快，也不影响电脑继续做其他工作。

## 为什么术语库重要

字幕质量很大程度取决于术语库。尤其是金融交易视频，如果术语翻译不地道，即使句子语法正确，观看体验也会很差。

例如：

- `order flow` 应译成“订单流”。
- `price action` 应译成“价格行为”。
- `scalper` 在交易语境里通常不是“黄牛”，而是“剥头皮交易员”或“超短线交易员”。
- `fair value gap` 更适合按交易圈常用说法译为 “FVG / 公平价值缺口”。

如果换到电影、法律、医学、游戏或产品教程，也应维护对应领域的术语、专名和表达习惯。

## 热词和专名需要持续维护

ASR 再好，也可能把专名、品牌名、人名、指标名拆错或识别错。例如交易平台、课程品牌、指标名称、人名和缩写，经常需要人工沉淀为固定规则。

维护时要先判断问题发生在哪一层：如果英文原文一开始就被听错、拆词或把人名品牌名识别错，这是 ASR 热词问题；如果英文已经听对，只是中文翻译不地道，这是术语库或质检规则问题。不要把所有翻译偏好都塞进 ASR 热词库。

建议每处理 3-5 个同领域视频，或一次大批量视频结束后，集中检查一次常见问题。反复出现的专名、平台名、指标名、ticker 和高价值短语，可以维护到阿里的热词表里；中文译法、字幕风格和句子节奏，则维护到术语库、修复规则和质检规则里。

## 后续维护重点

后续最重要的不是频繁更换模型，而是持续维护：

- 常见术语的标准译法。
- 人名、品牌名、平台名、指标名。
- ASR 经常拆错或识别错的词。
- 不同领域的表达风格。
- 正式交付前的质检规则。

只要这些内容持续积累，这套工作流会越来越接近对应领域的专业人工字幕效果。

## 常见问题 FAQ

### 我需要自己记命令吗？

不需要。把本地视频路径交给 AI，回答源语言、目标语言和是否启用屏幕上下文等确认问题即可。命令行是 AI 在后台执行流程的方式。

### 为什么流程中途会停下来？

正常情况下不应该停在分段翻译环节。v1.3.0 以后，`segments.txt` 默认由固定的 `qwen-mt-plus` helper 自动生成。只有环境、网络、API、QA blocker 或连续自动修复失败时，流程才会暂停并让 AI 根据错误信息处理。

### 什么时候需要启用 OCR/屏幕上下文？

只有当 PPT、图表、软件界面、代码、指标名或画面文字会影响理解时才建议启用。普通口播、访谈、播客、画面文字不重要的视频，建议关闭。

### QA 没通过是不是要我手动改？

不是优先让用户手动改。AI 应先根据 `final_qa_prompt.txt` 和 `final_qa_report.md` 自动修复，再重新导出。只有连续修复失败，或需要专业判断时，才应询问用户。

### 支持实时字幕或直播同传吗？

不支持。当前工作流只处理录制好的本地视频或音频文件。

### 支持把字幕硬压进视频吗？

不支持。当前输出 ASS/SRT 字幕文件。硬压视频属于后续独立步骤。

### 支持翻译成中文以外的目标语言吗？

架构可以扩展，但当前内置提示词、术语库、排版和质检规则主要围绕中文字幕优化。如果要稳定翻译成英文、法语、西班牙语、意大利语等目标语言，需要补充对应目标语言的规则。

### 音频质量很差会怎样？

低质量音频、强口音、多人重叠对话和背景噪声会明显降低转写准确性。重要内容建议人工抽查。

### 长视频为什么一直没消息？

不应该一直没消息。超过 10 分钟的任务，AI 应提前说明预计耗时，并每 10 分钟汇报一次当前状态。如果长时间没有反馈，可以让 AI 读取本次运行目录里的 `workflow_status.json`，说明当前卡在哪个环节。

### 长视频分段翻译很慢怎么办？

长视频的分段翻译默认由 `qwen-mt-plus` helper 执行，它支持缓存和续跑。生产默认串行调用，慢通常来自视频词数多、网络/API 排队、或 QA 修复耗时。AI 不应为了追求速度临时提高并发或回退到其他模型；如果遇到限流，应降速并从缓存续跑。也不应把任务丢到后台后说“等通知”；应在同一轮次里持续轮询，并按约定反馈状态。

## 不适合的用法 / 反模式

- 不需要自己记命令。把视频路径交给 AI 即可，命令行只是 AI 在后台执行流程的方式。
- 不要默认启用 OCR/屏幕上下文。只有 PPT、图表、软件界面或屏幕文字会影响理解时才启用，否则会增加耗时和成本。
- 不要把低质量音频当成完全自动化任务。强噪声、强口音、多人重叠对话会影响转写准确性，重要内容应抽查。
- 不要随意切换 ASR 模型。当前流程围绕 OkFile + 阿里 Fun-ASR 的词级时间戳设计，换模型需要重新验证时间戳和对齐效果。
- 不要把 QA blocker 直接丢给用户手动修。AI 应先根据 `final_qa_prompt.txt` 自动修复，连续失败或需要专业判断时再请用户确认。
- 不要把字幕硬压进视频。本工作流交付 ASS/SRT 字幕文件，硬压视频属于后续独立步骤。
- 不要让长视频任务长时间无反馈。超过 10 分钟的任务，应每 10 分钟报告一次状态。
- 不要临时手写并发翻译脚本。分段翻译固定走内置 `qwen-mt-plus` helper 的串行缓存能力，避免流程不可复现，也避免高并发限流导致模型回退。

## 字幕效果示例

```text
原文：Now, this process is very important, because once you have simulated how much money you can make into the year...
字幕：这个过程至关重要：一旦你模拟出全年潜在收益，再每天实盘交易，情绪干扰就会大幅降低
```

好的字幕不是逐词硬翻，而是在不偏离原意的前提下，让中文自然、信息密度合适、显示节奏舒服。
